# МАСТЕР-ДОКУМЕНТ: ML-система прогнозирования спроса (Demand Forecasting)

## 1. Введение и ссылки на графические модели

В документе используются следующие диаграммы (описание логики и ссылки на условные имена артефактов):

1.  **[D1] Модель бизнес-процессов (BPMN/IDEF0) "AS-IS" и "TO-BE"** — показывает переход от ручного интуитивного заказа к автоматизированному формированию заказов на основе ML-прогноза.
2.  **[D2] ER-диаграмма (инфологическая модель)** — демонстрирует структуру данных в хранилище (Star Schema для DWH).
3.  **[D3] Диаграмма архитектуры системы и распределенных данных** — показывает поток данных от OLTP-систем через DWH и Feature Store в ML-сервис и обратно в WMS.
4.  **[D4] Структурная UML-диаграмма классов/компонентов** — показывает ядро системы (Feature Engineering, Model Serving, Orchestrator).
5.  **[D5] Поведенческая UML (Диаграмма последовательности)** — иллюстрирует процесс инференса (Batch-прогноз) и триггеры для пополнения запасов.

## 2. Дизайн-документ (MLSD)

### 2.1. Цели и предпосылки

**Зачем идем в разработку продукта?**
Текущий процесс (BPMN AS-IS, [D1]) содержит высокую долю человеческого фактора. Менеджеры заказывают товары, ориентируясь на "глазомер", что приводит к кассовым разрывам. Продукт внедряется для централизации управления логистикой и построения системы **Predictive Replenishment**.

**Почему станет лучше от использования ML?**
ML позволит выявлять скрытые нелинейные зависимости (влияние праздников, погоды, дней недели, акций), которые человек не может обрабатывать в разрезе 10 000 товаров и 500 магазинов. Это снизит долю просрочки (Shrinkage) на 15-20% и увеличит уровень сервиса (In-Stock Rate) до 98%.

**Бизнес-требования и ограничения:**
*   Снижение операционных потерь (просрочка) на 20% за Q1 2027.
*   Увеличение валовой выручки за счет снижения Out-of-Stock.
*   Ограничение: система не может напрямую изменять заказы поставщикам без подтверждения (только рекомендательный режим на первом этапе).
*   Интеграция с существующей WMS (система управления складом) через REST API или шину данных (Kafka).

### 2.2. Требования

**Функциональные требования:**
1.  Расчет прогноза спроса на каждый SKU (Stock Keeping Unit) для каждого магазина на горизонт D+1, D+7, D+14.
2.  Формирование "Рекомендованного заказа" на основе прогноза + страховой запас (Safety Stock) + остатков на полке.
3.  Визуализация отклонений прогноза от факта (Dashboard для менеджеров).

**Нефункциональные требования:**
*   **Надежность:** Система должна выдавать прогноз даже при отсутствии части данных (Fallback-стратегия — использование среднего за 14 дней).
*   **Производительность:** Расчет прогноза для всех 5 млн комбинаций (SKU*Store) должен занимать не более 60 минут (ночной батч).
*   **Масштабируемость:** Горизонтальное масштабирование инференса.

### 2.3. Пилот и MVP

**Построение пилота:**
*   Выбрать 10 магазинов из разных кластеров (по размеру, региону) и 50 категорий товаров (скоропорт + FMCG).
*   Критерии успеха пилота: MAPE (средняя абсолютная процентная ошибка) < 15% для топ-100 товаров в этих магазинах. Сравнение с точностью ручного прогноза менеджера за предыдущий период.

**MVP vs Технический долг:**
*   *Входит в MVP:* Батч-инференс раз в сутки. Один базовый градиентный бустинг (XGBoost/LightGBM) с фичами по лагам и дням недели. Интеграция выдачи CSV/API.
*   *Технический долг (откладываем):* Real-time инференс (онлайн-корректировка в течение дня), использование LSTM/Deep Learning, автоматический подбор гиперпараметров (AutoML) и калибровка страховых запасов.

## 3. Методология решения

**Что делаем:** Задача **регрессии** (прогноз количественных значений продаж) и частично задача **классификации** (вероятность дефицита).

**Данные (Без чего не решить):**
1.  *Исторический спрос (Target):* Временной ряд продаж (штуки/кг) за 3 года (24 месяца). (Указанных 2.67 млрд записей достаточно).
2.  *Календарь:* Праздники, выходные, график завоза товаров в магазин.
3.  *Ценовой фактор:* История цен и акций (промо-коллаборации). Без этого прогноз искажается.
4.  *Внешние факторы:* Погода (особенно для хлеба и молока).
5.  *Остатки:* Данные об остатках на начало дня (для понимания, не было ли дефицита, искажающего историю продаж).

**Метрики качества (ML) и бизнес-связь:**
*   **Основная:** **MAE** (Mean Absolute Error) в штуках — напрямую влияет на количество закупаемого товара.
*   **Бизнес-метрика:** **FAST** (Forecast Accuracy) — % дней, где прогноз попал в доверительный интервал.
*   **SMAPE** (Symmetric Mean Absolute Percentage Error) — используется для борьбы с эффектом "нулевых" продаж. Связь с бизнесом: снижение MAPE на 1% дает экономию на логистике ~2%.

**Риски анализа:**
1.  *Искажение данных из-за Out-of-Stock:* Если товара не было на полке, мы не знаем, какой был реальный спрос. *Решение:* Импутация пропусков с использованием матричных методов (ALS) или моделей выживаемости.
2.  *Холодный старт (новые товары).* *Решение:* Использовать агрегацию на уровне бренда/категории для этих SKU.

## 4. Внедрение Production системы

### 4.1. Архитектура решения (Описание к [D3] и [D4])

Система состоит из следующих слоев:

1.  **Слой источников (Data Sources):** POS-терминалы (продажи), WMS (остатки), ERP (цены), внешние API (погода).
2.  **Слой хранения (Storage):**
    *   **Raw Zone (S3/MinIO):** Паркет/CSV исходных данных.
    *   **DWH (ClickHouse/Greenplum):** Хранит агрегированные витрины (продажи по дням). *Распределенный характер [D3] обусловлен огромным объемом — 2.6 млрд строк; данные шардируются по магазинам для параллельной обработки.*
    *   **Feature Store (Feast / Redis):** Хранит сгенерированные признаки для инференса (лаг-признаки, скользящие средние).
    *   **MLflow Model Registry:** Хранит версии моделей.
3.  **Слой вычислений (Processing):**
    *   **Airflow/Prefect:** Оркестрация (DAG: Extract -> Feature Engineering -> Train -> Predict -> Alert).
    *   **Spark / Ray:** Распределенный фиче-инжиниринг.
    *   **Kubernetes (Kserve/Seldon):** Поднятие моделей для инференса (масштабируется горизонтально в зависимости от числа магазинов).
4.  **Слой доставки (Action):** Backend API, отправляющий итоговый файл заказа в WMS.

### 4.2. UML-диаграмма классов (Компоненты) [D4]
*   `DataPipelineOrchestrator` — управляет задачами.
*   `FeatureGenerator` — принимает `StoreSKUContext`, генерирует матрицу признаков (Time-lags, Rolling stats).
*   `ModelTrainer` — реализует интерфейс `Trainable` (XGBoost, Prophet, Ensemble).
*   `Predictor` — запускает `predict()` на кластере.
*   `ReplenishmentCalculator` — применяет бизнес-правила (минимальный заказ, кратность упаковки) к сырому прогнозу.

### 4.3. Поведение (Диаграмма последовательности [D5])
1.  **Триггер:** По расписанию (02:00 AM) Airflow запускает DAG.
2.  **Действие:** Spark читает данные за последние 30 дней из ClickHouse.
3.  **Feature Store:** Извлекает из Redis предыдущие вычисленные признаки (для построения лагов).
4.  **Model Serving:** Отправляет batch-запрос в кластер моделей (параллельно для 500 магазинов).
5.  **Результат:** Модель возвращает `predicted_demand`.
6.  **Бизнес-слой:** Сервис вычисляет `Order = max(0, predicted_demand - current_stock + safety_stock)`.
7.  **Callback:** Отправляет финальный файл в очередь Kafka для WMS.

## 5. Инфраструктура и масштабируемость

**Выбор инфраструктуры:**
*   **Cloud (AWS/GCP) или On-prem:** Рекомендую **Kubernetes + Istio** для управления микросервисами.
*   **Почему распределенное хранилище ([D3]):** Из-за территориальной распределенности магазинов и объёма данных (2.67 млрд записей). Хранить это на одной машине невозможно по времени доступа. Используем **S3** для исторических данных (холодное хранилище) и **ClickHouse** для горячих данных (OLAP-запросы на лету).

**SLA и RPS:**
*   **SLA:** Доступность 99.9% во время ночного батча (с 2:00 до 5:00). Допустимый даунтайм — не более 5 минут (перезапуск пода).
*   **Latency:** Для пакетного режима не критична (обработка всего датасета за 45 минут). Для будущего онлайн-режима (если внедрим) — Latency < 200ms на запрос.
*   **RPS:** ~50 RPS (при запросе прогноза для одного магазина через фронтенд).

## 6. Риски и неопределенности

| Риск | Вероятность | Стратегия митигации |
| :--- | :--- | :--- |
| **Дрейф данных (Data Drift)** | Высокая | Мониторинг распределения признаков (Evidently AI). Ежедневный пересчет метрик на тестовой выборке. |
| **Изменение ассортимента** | Средняя | Введение категорийных эмбеддингов для "холодных" SKU. |
| **Проблемы с качеством POS-данных** | Высокая | Pipeline валидации данных (Great Expectations) — проверка на дубли, отрицательные продажи (возвраты). |
| **Нежелание менеджеров доверять системе** | Средняя | Внедрение "режима советчика" (не автозаказ), где менеджер видит прогноз и может скорректировать его "навесом" (коэффициент коррекции). |
| **Отказ внешнего API (Погода)** | Низкая | Fallback на климатическую норму (среднее за 5 лет). |

## 7. Детальное описание диаграмм (Текст для построения)

### [D1] BPMN AS-IS (Было)
*   **Старт:** Пришло утро.
*   **Действие 1:** Менеджер обходит склад (визуальный осмотр).
*   **Действие 2:** Менеджер открывает Excel/бумажный журнал.
*   **Действие 3:** "Прикидывает" кол-во, ориентируясь на прошлый вторник (субъективно).
*   **Ворота:** Ошибка при оценке -> Ветка 1: Заказ слишком мал (Потеря выручки); Ветка 2: Заказ слишком велик (Списание).
*   **Финал:** Отправка заказа поставщику.

### [D1] BPMN TO-BE (Стало) - https://github.com/Yanochka123/Project_ML/edit/main/BPMN_diagram_1.png
*   **Старт:** Система в 3:00 собирает данные.
*   **Действие 1:** ML-Сервис рассчитывает вероятностный прогноз (P10, P50, P90).
*   **Действие 2:** Business Rules Engine (BRE) накладывает ограничения (минимальная поставка, срок годности).
*   **Действие 3:** Система генерирует "Карточку заказа" с пояснением: *"Если закажете 150 ед. -> вероятность дефицита 5%".*
*   **Ворота:** Менеджер подтверждает ИЛИ автоматически отправляется (в зависимости от настройки).
*   **Финал:** Заказ интегрируется в WMS автоматически. (Исключен субъективный осмотр).

---

### [D2] ER-диаграмма (Инфологическая модель)
*   **Таблица фактов:** `sales_fact` (суррогатный ключ, `store_id` (FK), `product_id` (FK), `date_key` (FK), `quantity_sold`, `revenue`).
*   **Измерение (DIM) `store`:** `id`, `region`, `store_type`, `square`.
*   **Измерение (DIM) `product`:** `id`, `sku`, `category`, `brand`, `shelf_life_days`.
*   **Измерение (DIM) `calendar`:** `date`, `is_holiday`, `day_of_week`, `is_promotion`.
*   *(Связи: Звездообразная схема. sales_fact ссылается на все 3 измерения).*

---

### [D3] Архитектура распределенных данных
*   **Блок 1 (Edge):** POS-системы Магазинов (push данных раз в час).
*   **Блок 2 (Streaming):** Kafka Cluster (Топики: `raw_sales`, `raw_inventory`).
*   **Блок 3 (Storage):** Распределенный HDFS/MinIO (партиционирование по году/месяцу).
*   **Блок 4 (Compute):** Spark Cluster (обрабатывает партиции параллельно) -> генерирует признаки в Feature Store (Redis Cluster, шардирование по ключу `store_id`).
*   **Блок 5 (Inference):** GPU/CPU Pool в Kubernetes (поскольку моделей много — каждая модель может отвечать за свою категорию товаров, запускаясь в отдельном поде).

---

### [D4] UML Компонентов
*   **Компонент "Data Lake Connector"** -> Интерфейс `IDataReader`.
*   **Компонент "Feature Engineering"** -> Зависит от "Data Lake" и "Cache".
*   **Компонент "Model Registry"** -> Управляет артефактами.
*   **Компонент "Inference Engine"** -> Реализует `IPredictionService`. Имеет порт `REST (8080)` и `gRPC (9090)`.

---

### [D5] UML Sequence (Batch-процесс)

```
User/Airflow -> Orchestrator: triggerPredict()
Orchestrator -> DataWarehouse: getSalesHistory(store, sku)
DataWarehouse --> Orchestrator: RawData (Parquet)
Orchestrator -> FeatureEngine: generateLags(Data)
FeatureEngine --> Orchestrator: FeatureVectors
Orchestrator -> ModelServer: predict(features)
ModelServer -> ModelServer: loadModelFromRegistry()
ModelServer --> Orchestrator: PredictedDemand
Orchestrator -> WMS_API: sendReplenishmentOrder()
WMS_API --> Orchestrator: OrderAccepted
Orchestrator -> Dashboard: UpdateMetric(MAPE)
```

---

**Заключение:** Предложенная архитектура решает задачу масштабируемости за счет четкого разделения слоев (озеро данных -> Feature Store -> ML) и использования распределенных вычислений. MVP сфокусирован на Batch-обработке, что позволяет быстро получить бизнес-ценность (экономия закупок), а технический долг (Real-time) будет погашен во второй итерации после сбора обратной связи от бизнеса.
